# 理念

从理论 -> 实践本身，也应当维护好文档主线。也就是要利用 docwarden 开发 docwarden，
和 https://github.com/openai/symphony 的实践一致

在定义实践前，首先应该完善本项目的 ai infra

在这里定义一些要点：

## 最终实现形态

codex plugin（skills族） + CLI 形式。为什么是 cli？在此之前已经设计过一版的 docwarden，最终的形态是 skills 各自负责各自的 python 脚本，这些脚本实际上服务的都是同一件事，及 docwarden 的实现。孤立的脚本无法模块化的组织和复用，扩展到 cli 形式则是顺势而为。

## self-loop 迭代

所有的 ai 基础建设都应该能够自迭代。

1. skills 需要不断迭代才会变得更好，需要重设计官方的 skill-creator，以更严格的规范和约束设计 skills
2. 需要有 improve skill 的 skill，显示进入优化 skills 的 workflow

## 文档编写准则

区分原始文档和 ai-generated docs。docs 目录下全部都是人类编写、维护的文档，目前只有作者 sayoriqwq

## 代码与docs link

做 task 的时候不能忘记同步文档更新，需要建立显示追溯机制。无论是在 AGENTS.md 里编写，还是 skill workflow，又或者是脚本。评估这里实现效果的要义是：新 agent 在没有获取任何额外上下文的情况下，能正确的找到 docs，并且按照它的任务更新 docs

## 文档最小化和断言

明确 ai 建立后的文档，需要满足这样的约束：

1. 一份 md 只负责一件事情。单一职责原则
2. md 文档的存储位置（文件夹结构），和命名必须有意义且细致。服务于 agent 搜索
3. 若状态需要持久化存储，且是文件级别，采用 front-matter 形式
4. 接着2,文件夹结构需要有 trace 机制，则需求一个 roadmap.md 文档，唯一作用是指路。告诉来到这的 agent：这个文件夹下的 docs 在做什么，有哪些，怎么找
5. 若文档需要结构化，需要先定义 template，后续创建采用脚本去创建，模型在 template 的基础上修改，绝对不允许模型自由发挥

6. 祈使句提示词的约束：

https://datatracker.ietf.org/doc/html/rfc2119

参见 rfc 规范，learn from https://github.com/openai/symphony

强约束： MUST/MUST NOT

次强约束：SHOULD/SHOULD NOT

模型发挥： MAY

## skills 组织规范

参考 https://github.com/openai/symphony/tree/main/.codex/skills，这里有着一系列的 skills，一起服务于同一个 workflow

## 定位

这是 docwarden 之上的一层约束，反应 docwarden 的分层管理文档的模式。
在开发 docwarden 的同时，一并维护这里的信息，后续可能会基于在这里的实践，把基建扩展到 user 层级，而不是只服务于当前项目。

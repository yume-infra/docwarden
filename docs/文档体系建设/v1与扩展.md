# v1与扩展

当前理论约束下，能得到：

1. 体现项目规范和稳定基线的体系化信息，可以用于派生 skills 或者给 agent 提供稳定上下文
2. 规范化的迁移过程，即在 review 最少内容的情况下（stable），输出与用户意图偏移尽可能少的文档

仍缺少：

1. 在做需求时始终保持跟进进度的 plan 文档（workflow 相关，working 从哪来？）。这里刻意不做。
2. 执行层面的规范、评估标准与失败模式（执行 workflow 相关，task 从哪来，如何管理，如何规划，以及并发考虑）
3. 如何从 for agents 的 stable 文档中派生出适合 user 阅读的表达层（agent 文档编写相关）
4. 距离无负担的清理仍有距离。是否需要维护一个 origin/archive 层级？这里其实有点纠结

在此之外，还有另外的体系建设 idea，以完成整个 workflow 的闭环。

1. 了解业务 -> 了解用户。把过程文档中能够辅助 agent 了解用户画像的信息提炼出来，从而实现更好的定制化的体验。如编码风格
2. 项目文档 -> 个人 wiki。在项目开发中，给出个人的 agent 知识库维护，理念参考 llm-wiki，基于个人的技术 wiki，可以实现更多的事情：复习、检测、提炼（比如派生周报）
3. 自然语言 -> agent 理解。目前而言编码模型基本都是国外的，中 -> 英已经产生了一次偏移，核心的 prompt 尽可能用英文写。但是语言的匹配/转换这些内容还是无法稳定。有没有一些核心的词语可以稳定 agent 的执行路径呢，而不是靠一些自己都不知道有没有用的标识符： ⚠️、**加粗**、祈使句式。目前唯一已知的是重复管用。

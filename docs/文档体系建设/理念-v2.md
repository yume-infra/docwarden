# 文档体系理论 v2

> ⚠️ ai warning
>
> 本文由 Codex 根据 sayori 在 `.docwarden/task/03-ai-infra-min-schema/` 与 `.docwarden/task/04-task-review-spec-pipeline/` 中已经 review 通过的讨论结论起草。
> 在 sayori 完成 review 前，本文只代表对当前理论的整理，不应被视为最终权威版本。

## 一、v2 修正了什么

v1 的核心判断仍然成立：文档是 code agent 获取项目语境的重要来源，文档体系的目标是减少反复 prompt、减少理解偏移，并让项目中的稳定认知可以被持续维护。

但 v1 使用 `working / stable` 作为主模型。继续推进后发现，`stable` 这个词把几个不同责任混在了一起：

1. 任务推进过程中的短期材料。
2. user 需要 review 的最小审查单元。
3. agent 执行时需要读取的规范层。
4. user 理解项目时需要阅读的叙事层。
5. 长期沉淀、可链接、可查询的知识网络。

这些东西都需要“稳定”，但它们不是同一种稳定。

因此 v2 不再把 `stable` 当作核心层级，而是把文档维护流程修正为：

```text
task -> review system -> spec
                    -> guide
                    -> wiki
```

这个模型表达的是：

1. `task` 捕获任务过程和上下文。
2. `review system` 从 task 中提炼出 user 可审查的 review surface。
3. review 后，模型把内容分发到 `spec`、`guide`、`wiki`。

## 二、项目边界：docwarden 与 contexta

v2 需要先明确一个产品边界。

`docwarden` 负责操作流程协议。

它回答：

- agent 在哪里创建任务材料。
- agent 如何记录计划、草案、log 和 review 过程。
- agent 什么时候能改 `.docwarden`。
- agent 什么时候必须停下来等待 user review。
- review 后的内容如何进入后续层级。

`contexta` 负责文档内容格式协议。

它回答：

- 文档内容的 metadata 如何设计。
- 断言、语义单元和模板如何表达。
- 自然语言如何更稳定地转成 agent 可执行的结构。
- 文档正文如何被 lint、解析和约束。

换言之：

- 操作流程协议属于 `docwarden`。
- 文档内容格式协议属于 `contexta`。

这一区分很重要。否则 docwarden 很容易过早进入断言 schema、template grammar 或 semantic lint 的设计，从而把流程层和内容层搅在一起。

## 三、docs 的位置

当前项目里的 `docs/` 是 human-maintained source layer。

它的特点是：

- 由 sayori 编写和 review。
- 更偏向向 user 描述理论。
- 是当前最可信的理论来源。
- 不是已经完成 agent-facing 表达优化的 spec。

这意味着 agent 可以读取 `docs/` 作为判断依据，但不能把 `docs/` 等同于最终 agent 执行规范。

当对话中产生了可能进入 `docs/` 的内容时，默认应该先进入 `.docwarden/task/`、review notes 或其他 task material。只有当 user 明确要求编辑具体 `docs/` 文件时，agent 才能直接修改 `docs/`。

## 四、task：短命任务过程层

`task` 是短命任务过程层。

它承载：

- 上下文捕获。
- 计划。
- 草案。
- log。
- review 过程记录。
- 临时判断。
- 任务推进材料。

旧理论中的 `working` 已经被 `task` 替代。

原因是 `working` 更像一种状态，而 `task` 更明确地表达：这里的材料围绕一个具体任务存在，生命周期较短，不应该承担长期知识沉淀的职责。

因此 `.docwarden/working/` 不再保留，当前项目中已经迁移为：

```text
.docwarden/task/
```

一个 task 可以有自己的 `index.md`、`plan.md`、`log.md` 和若干草案文件。

其中：

- `index.md` 是工作面 lead file，用于告诉 agent 这个 task 在做什么、应该从哪里开始。
- `plan.md` 记录本轮推进结构。
- `log.md` 按时间追加记录发生过的 workflow 事件。

task 里的文件状态可以先保持最小集合：

- `draft`：agent 已生成，等待 user review。
- `reviewing`：正在 review，或正在根据 review 修改。
- `accepted`：user 接受为当前 task 定义。

`accepted` 不表示进入 spec、guide 或 wiki，也不表示已经成为长期基线。

## 五、review system：打开黑盒的审查系统

v1 已经确认：人工 review 目前不可替代。偏移无法消失，只能减少。

v2 对 review 的修正是：review 不是一个 Markdown 文件，也不是 spec 的 user 版本，而是一个系统。

review system 的职责是从 task 中提炼 user 需要审查的内容，降低 user review 成本，让 user 能够在最小成本下判断、纠偏和确认关注点。

review 的关键不是“把所有产物都给 user 看一遍”，而是尽可能组织出最小有效审查单元。

一次 review 应该解决：

- 当前真正需要 user 判断的问题是什么。
- 模型是否理解错了关键概念。
- 哪些关注点必须保留。
- 哪些内容应该进入 spec、guide 或 wiki。
- 哪些内容应该放弃或延后。

review 应当尽量只做一轮。

user 在这一轮中解决问题、确认关注点、纠正偏移。之后模型应当自己提炼内容，并分发到后续层级。如果 user 觉得分发结果不对，可以再去目标层级 review，而不是让 review 层长期承担理解入口。

## 六、review surface：短命界面，长期留痕

review system 需要产生 review surface。

review surface 是给 user 审查用的界面材料，可以是：

- Markdown。
- HTML。
- JSON 驱动的界面。
- 对比视图。
- 临时 review data。

HTML 是重要候选形态。相比 Markdown，它更适合承载：

- 折叠内容。
- 对比视图。
- 状态标记。
- 颜色和视觉分区。
- 表格和卡片。
- 主结论、风险、待确认问题、来源上下文的分区展示。

但 review surface 本体应该短命。

它不是长期知识层，也不是最终展示层。它的价值在于降低某一轮 review 的成本。

review 后需要保留的是 decision trace，而不是完整保留审查界面。

decision trace 可以记录：

- 哪些问题被 user 确认过。
- 哪些关注点被确定。
- 为什么某些内容进入 spec、guide 或 wiki。
- 哪些内容被放弃或延后。

decision trace 的具体落点可以是 task 的 `log.md`、目标文件的 source/provenance，或 wiki 中具有长期价值的 ADR / decision 页面。具体格式后续需要单独设计。

## 七、spec：agent-facing 执行规范层

`spec` 是 agent-facing 的执行规范层。

它服务 agent 编码和项目执行。

spec 应该比 task 更稳定，比 review 更面向执行。它不追求面向人类的完整解释，而是追求让 agent 在新会话中快速恢复正确约束。

spec 适合表达：

- 项目级规则。
- agent 操作边界。
- 代码或文档维护流程。
- 明确的正例和反例。
- 失败模式。
- 必须遵守的执行约束。

spec 不等于当前 `docs/`。

`docs/` 可以是理论来源，spec 是经过 review 和表达优化后的执行入口。

## 八、guide：user-facing 叙事层

`guide` 是 user-facing 的叙事层。

它服务：

- user 理解项目。
- 新需求启动时恢复上下文。
- 把分散的理论、决策和知识组织成线性路径。

guide 不是 wiki。

wiki 适合小颗粒、可组合、可链接、可查询；guide 更适合连续阅读和建立全局理解。

这类似代码中的模块拆分和入门导览之间的区别：模块拆分有利于维护和组合，但新手进入项目时仍然需要一条更线性的理解路径。

因此 review 后不仅可能产出 spec，也可能产出 guide。

当 user 需要理解项目为什么这么设计，或者需要基于现有项目提出新需求时，guide 会比 spec 和 wiki 更友好。

## 九、wiki：长期知识网络

`wiki` 是长期知识网络。

它承接 `llm-wiki` 的思路：让 LLM 持续维护一组可链接、可组合、可查询的 Markdown 文件，而不是每次查询时都从原始材料重新检索和拼接。

wiki 适合沉淀：

- ADR。
- 纠错经验。
- 用户偏好。
- 可学习内容。
- 项目理解材料。
- 关联概念。
- 被多处复用的小颗粒知识。

wiki 的价值在于复利。

每次任务、review 和查询产生的稳定知识，都可以被整理进 wiki。后续 agent 或 user 不需要重新从长对话和过程材料中挖掘，而是可以直接沿着链接和索引查询。

wiki 不负责提供完整叙事。

wiki 像维护良好的模块化代码：结构清楚、引用丰富、利于组合。但对第一次进入项目的人来说，过度跳转会增加理解成本。因此 wiki 需要和 guide 分离。

## 十、promote 与 pick 的重新定位

在 v1 中，`promote` 可以理解为 working -> stable 的主路径，`pick` 是从 working 中捡出 stable 不关注但仍然有价值的 side 内容。

在 v2 中，`stable` 被拆解后，这两个词需要重新放置。

更准确的理解是：

- `promote`：从 task 经 review 后，进入 spec / guide / wiki 中的主产物路径。
- `pick`：从 task 中提取 promote 主路径不关注、但仍有长期价值的 side 内容。

pick 不是 review 选择动作。

pick 也不是自动进入 `docs/` 的流程。

pick 可以服务 wiki、ADR、纠错经验、用户偏好、可学习内容等长期建设。由于 task 是短命层，pick 产物不应该长期留在 task 中。

promote / pick 的触发时机和完整编排仍属于后续 workflow 设计，本文件不展开。

## 十一、最小目录结构

当前 `.docwarden/` 的最小结构是：

```text
.docwarden/
  task/
    index.md
  review/
    index.md
  spec/
    index.md
  guide/
    index.md
  wiki/
    index.md
```

目录含义：

- `task/`：短命任务过程层。
- `review/`：review system 的 artifact 存放层。
- `spec/`：agent-facing 执行规范层。
- `guide/`：user-facing 叙事层。
- `wiki/`：长期知识网络。

每层入口统一使用 `index.md`。

统一入口名可以降低 agent 查询成本，也更符合 agent 在目录中寻找入口文件的习惯。

## 十二、v2 当前仍未解决的问题

v2 只是修正了主模型，还没有完成完整系统设计。

仍需后续展开的问题包括：

1. review system 的详细设计。
2. HTML review surface 的最小可用形态。
3. review artifact / decision trace 的 schema。
4. spec / guide / wiki 的最小产物模板。
5. promote / pick 的触发和编排。
6. contexta 如何定义断言、metadata 和内容格式。
7. 如何从 task 中稳定分发到 spec / guide / wiki。
8. 如何评估 spec 是否真的能服务 agent 执行。
9. 如何评估 guide 是否真的能服务 user 理解。
10. 如何维护 wiki 的链接、索引、冲突和过期信息。

## 十三、当前结论

文档体系 v2 的核心变化是：不再把所有稳定产物压进单一 `stable` 层，而是承认文档系统至少包含任务过程、审查系统、agent 规范、人类导览和长期知识网络五种不同责任。

因此当前理论主干是：

```text
task -> review system -> spec
                    -> guide
                    -> wiki
```

其中：

- task 负责捕获过程。
- review system 负责打开黑盒、降低 review 成本。
- spec 负责约束 agent 执行。
- guide 负责帮助 user 理解。
- wiki 负责长期知识复利。

这不是最终完整实现，而是下一阶段系统设计的理论地基。

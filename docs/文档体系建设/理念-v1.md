在高强度用 claude code 和 codex vibe coding 了几个月后，我意识到**文档**比我想象中重要的多得多。文档是代码之外，coding agent 能获取的主要项目语境源，它承载代码无法表达的：业务意图、决策理由、用户偏好等。好的文档体系可以让 agent 写出更符合项目规范的代码。

但在维护文档的过程中，其实有相当多的问题。文档这个东西，创建容易迭代难。过期的、错误的文档会导致 agent 误判。因此有必要在代码推进的同时，维护一条文档推进流。

下文会探讨一份文档体系理论，实现这个愿景。

## 一、定义与目标

### 理念
和主流的 workflow，如 superpowers、gsd、omx 等不同，在这里**只负责和文档处理的一切**。也就是说最后实现的结果是 workflow 无关的，你可以在任何地方尝试使用它去组织文档。

### 文档
md 文档即组件，可组合、可复用。是最小**组织单元**

文档的构成：metadata 和**结构化**正文。

文档的位置：目录和名字是查询规范，需要让 agent 能够通过有意义的命名规范以快速定位文档。

metadata：用于文档级别的持久化状态信息。

结构化正文：使用**脚本**稳定约束结构，而不是完全依赖模型自由发挥。目的在于提供后续可持续迭代的规范。

### 断言
文档迭代时被识别和审查的语义单位。是最小**语义单元**

一份 md 文档由多条断言组成

示例：Next.js 项目的一句约束：

本项目使用 Next.js 16，相关 API 可能不同于模型训练数据，需要读取本地 `node_modules/next/dist/docs/`

### 目标与基线文档
目标：实现一套文档维护机制，在推进日常工作的同时，利用

1. agent 的当前会话记忆
2. workflow 产出的过程性文档

产出能够描述当下稳定认知的**基线**文档。

基线文档作为 agent 默认上下文入口，从而减少反复与 agent 沟通和纠偏的过程。实现的效果：迭代了一段时间后，产出一个 stable 文档目录。在做新需求时，agent 会从这里获取所需的描述性信息，更好的理解**业务**、**规范**。

### 基线的定位
文档很显然不能作为项目事实的唯一真源。它只负责补充 agent 认知的默认基线。

当文档与代码或是运行行为冲突的时候，需要有明确的优先级和审查机制来处理冲突，这也是迭代的一部分。

### 差异
面向 vibe coding 而非 harness。目前理论仍停留在人工审核是**必要**的阶段，下一步方向仍不明确。

## 二、为什么需要文档
code agent 工作时，对项目理解的一切都存在于当前仓库中。清晰、意图明确的代码可以对 agent 起到良好的指导作用，除了代码，还有注释、文档这些补充信息。

有说法是良好的代码不需要注释。诚然，好的代码自己就是说明书，尤其是类型完备、组织良好的代码。

但在实践中，尤其是一些“看起来不那么优雅”的代码，补充必要的注释相当重要。实际开发中充满了各种不确定性和妥协，总是不能做到模型理解中的“最佳实践”，agent 会有自己的一套标准：这也不行，那也不行。但这是我们项目需要的。如果利用 agent 解决后不留痕迹，后续迭代必有坑。

因此需要文档来做辅助。上述情景需要的是一个补充业务描述的文档（或者说 adr），告诉模型为什么这么做。

## 三、文档的受众
文档是给谁看的，这一点大部分人都忽略，实际上至关重要。

给 agent 做的文档强调绝对的规范和边界，需要 guide 作用。评估：当 clear 所有上下文后，新 agent 能否通过这个文档快速的恢复之前的上下文状态，**且没有大方向偏移**。

给人类做的文档强调绝对的可读性，需要简洁、描述性。评估：人看不看的懂，看的费不费劲。

同样举个例子：

代入到技术文档的角度。比如在 Next.js 的官方文档，对概念的进一步解释、为什么这么做，api 怎么用的描述性语言，都是面向人类的，让开发者能看懂，学得会。

此时转变思维方向，给 agent 的实际上只需要四个东西：

1. api 文档（源）
2. api 在什么时候用（条件）
3. 使用该 api 需要在什么环境下（边界）
4. 关键正/反例（参考）

其他内容对 agent 来说意义就不大了，过多的杂项信息完全是污染上下文。

实际上这也是 llm.txt 做的事情。docwarden 就希望提供类似的一种规范，只不过针对 md。

## 四、文档的生效范围
环境即文档生效的范围。用一个最简单的 AGENTS.md 的例子来说，就是：越靠上的规则，越应该抽象；越靠近项目的规则，越应该具体。

接下来用一些例子详细的说明。

> 回复默认使用简体中文

这显然是一个最泛、最通用的规则，其应该处于根 AGENTS.md。

> 禁止静默回退，**切勿使用回退值来掩盖本不应缺失的数据**

codex 非常喜欢写各种 normalize 函数，以及回退值以通过整个模块。这是限定给 codex 的规则。

```md
<!-- BEGIN:nextjs-agent-rules -->

# This is NOT the Next.js you know

This version has breaking changes — APIs, conventions, and file structure may all differ from your training data. Read the relevant guide in `node_modules/next/dist/docs/` before writing any code. Heed deprecation notices.

<!-- END:nextjs-agent-rules -->
```

这是 Next.js 16 最新提供的一份 rules，这里的抽象程度减弱，明确了技术栈：Next.js 16，在这个环境下，这个规则通用。

> Next.js-i18next-best-practice.md

进一步的靠近项目，这份文档可以是一个 skill 的基线文档，用于指导有 i18n 需求的 Next.js 项目的代码编写。

希望这些例子能帮助你理解到我想表达的意思：

全局 -> 项目的抽象应该集中在**项目环境**这个点上，如果某条件需要在满足特定环境才应用，就应该被下放到项目，反之越抽象的 skill 成立条件越宽，跨项目大概率会有复用需求，就应该提升。

生效范围不仅由项目决定，也受用户、agent、技术栈、模块和任务阶段影响。

## 五、偏移及其应对
我在这个月初的时候，还坚定的认为 skills 只能人工写，在那时主要担心在迭代过程中的偏移问题。

利用当前的创建 skill 的 skill，结构上可以通过校验，内容上却永远不能保证内容 100% 符合预期。

这其实就像薛定谔的猫一样：只要人不去观测它，就永远无法消除这里的不确定性。甚至说人介入其实也会有理解上的偏移，因为自然语言 -> agent 理解的结构的这一环，不是使用这个工具的我们能掌控的。

偏移导致的结果就是 agent 决策走偏，结果不符合预期。我有时也会全程监督 agent 干活，展开它的思考过程，看看它调用的脚本，在他思考的方向出现偏移时立刻打断。其实 harness 就是想解决这个问题：如果监督它的不是人，而是在环境中自己就能走回来呢？

基于这个 case，我得出了两个结论：

1. 人工 review 目前仍绝对不可替代。也就是必须要存在一个打开黑盒的过程。
2. 偏移无法消失，只能减少。

由这两个结论的推论：

1. 如果所有内容都强制由人介入，实在是太慢了。但有没有可能只 review 一小部分？显然是可行的。需要一个环境和工作流，让 agent 能稳定产出可用的文档，我们只 review 其中的一小部分。
2. 需要对一份相对稳定的文档继续迭代，尽可能的消除偏移。

## 六、文档状态与迭代机制

为此，我引入了两种文档状态：**working** 与 **stable**。

- **working**：维护过程中的计划与上下文。由于模型的计划依赖上下文机制，需要将其思考内容做成短期记忆，方便审查和 handoff，所以用 working 目录维护。
- **stable**：当计划做完，尤其是经历几个版本后，老文档会不可避免地出现口径不一致的问题。此时引入 stable 层级，在这里映射 working 中对应的业务内容，但做精简和约束，只保留当前稳定的基线。核心是确定一个基线，当 agent 工作时发现不一致，就去这里查找信息做参考，免去人工 prompt 的行为。

对 stable 的内容仍需要审查。这是当前理论的局限。但这里的审查会起到比预期有用的多得多的效果。因为 prompt 质量参差不一，在人与人之间交流的时候都有信息的丢失，和模型间更是如此。在我的实践中，我在 review 里找到了相当多的口径不一致问题，这些问题有的反哺了我对项目的理解，让我更进一步理解自己的需求，有的让我第一时间找到模型理解的偏移点，及时纠偏。

### stable 的迭代
当实现了某些涉及到改变基线的改动时，需要一个稳定的迭代机制以减少偏移。

要实现迭代的稳定：

1. 基线需要最大限度的分清楚职责，且规范明确。这样就可以有精确的修改目的地。
2. 迭代规则本身的 guideline **必须边界清晰**。也就是说，不论内容提炼的纯度和准确性，至少不会乱迭代。

在迭代之后，需要审查。

### working 的时效与 pick
working 的内容具有时效性。经历了一段时间后，上下文已经断了。如果想要保证原始的信息不丢失，还需要引入一个 pick 过程：由 agent 去对 stable 文档和 working-on 文档做一次 diff，然后提炼出在 stable 中已经隐去的信息，暴露给 user 做一次显示的 pick。

在这个环节中，最重要的点是**模型**对文档的理解能力。不要有重复的，也不要有遗漏的。这里的设计还没有一个比较好的定论。模型如何理解语义？尤其是对中文而言。现在只解决了核心的动词部分，使用 https://datatracker.ietf.org/doc/html/rfc2119 的短动词结构。

但总而言之，引入这个环节后，就可以有依据的清理掉 working-on 中的内容了。

同时，working-on 中大部分都是决策变量和临时环境，这些辅助性内容在完成任务后变成对当前需求毫无意义的杂质。然而杂质也有其价值：某些 adr 实际上体现了 user 的一些隐式的规范和约束，即用户偏好；临时环境中也隐藏着可扩展性。如果想要最大限度的利用这些信息，就需要在 pick 过程中，把这些内容以最上面所述的抽象层级给出一个答案，从而提炼到稳定文档或用户知识库中。
